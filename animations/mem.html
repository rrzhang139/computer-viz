where we are: a table of words

Memory doesn't know whether it's holding instructions or data — it just stores addressable words. Give it an address, it hands back the word living there. That's the whole contract, and it's why this block is plain mem: the same hardware is the instruction memory on the fetch path and the data memory on the load/store path.

On the left is the memory as a table: four 8-bit words at addresses 00, 01, 10, 11 (seeded here with a little program). On the right, a read MUX — the very multiplexer you already built — picks one. Toggle addr1/addr0 and watch the addressed row light and ride out as rdata.

address picks a row

Click addr0 so the address is 01.

The MUX's select flips and the second word lights up — its bits flow across into the MUX and out as rdata. Reading is purely combinational: no clock, the word appears the instant the address settles. An address is just a row number in binary: 00→row 0, 01→row 1, 10→row 2, 11→row 3.

Walk the address through 00, 01, 10, 11 and watch the lit row slide down the table.

Four words, one address, one selected — that's memory. The other three words sit untouched; only the addressed one is read.

the read MUX is the picker

Hover the read MUX and hold still.

It swaps for the real /mux.html — the same 4-to-1 multiplexer, here choosing among the four stored words instead of four loose bits. The address drives its select lines; the decoder inside raises one, opening exactly one path so the addressed word flows through. (Click to drill all the way in.)

A real memory is this, vastly bigger — millions of words, the address widened to dozens of bits driving a tree of MUXes — plus a write port (an address decoder gating the clock to one word) so you can store as well as load. We'll add that port when we reach loads and stores; for fetch we only ever read.

why this matters

This is where the program lives. Point the address at a row and out comes a word — and when that word is an instruction, this block is the instruction memory on the fetch path. Feed it the program counter as the address, and each cycle it hands back the next instruction to run.

Same block, different job: address it with an ALU result instead and the words it returns are data — that's the data memory a load reads or a store writes. One mem, two roles. Next we wire a program counter to its address input and let it step through the program — that's fetch.
