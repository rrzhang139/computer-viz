where we are: the whole loop, assembled

Every page until now built one part in isolation. This page is the assembly: it takes two of those parts — the register file on the left (where the numbers live) and the ALU on the right (where the math happens) — and wires them into the exact loop a real CPU runs on every instruction.

Over the next steps you'll execute one instruction end to end, one button at a time: add x3, x1, x2 — "read register x1, read register x2, add them, store the sum into x3." That's four moves — read, read, operate, write-back — and you'll do each by hand, then read your own result back to close the loop.

Press ↺ reset so the values match what's described, then press next →.

read the map: left → right, then back around

Orient first. Every input is a labeled terminal on the LEFT edge. The two read-address lines raddrA1/raddrA0 choose the first source register — you drive them with the read x1 buttons a1/a0; raddrB1/raddrB0 choose the second (read x2 → b1/b0); waddr1/waddr0 pick the write target (write x3 → w1/w0), with we to enable the write, clk the clock, and op1/op0 the operation (the op buttons). The two big blocks are the register file and, fed by it, the ALU; the answer leaves on the RIGHT as result (with the carry-out Cout above it). The two wires between the blocks are the whole point — the file's two read outputs cross into the ALU as its operands, and the ALU's result loops back along the bottom into the write port.

Hover the register file block and hold still.

The box swaps for its real internals, lit to this exact state: r0 glows (the one seeded 1), and read port A's MUX is already forwarding r0's bit — that lit path out the right edge is the 1 sitting in operand A. The write-side decoder is dark, because we = 0: nothing can be written yet.

Now hover the ALU block.

Inside, the full adder and the AND/OR/XOR gates are all chewing on A and B at once; the op-MUX at the output is forwarding just the ADD line. What you see on hover is exactly the page you'd land on by clicking — click either block any time to drill all the way down.

read your two source registers

Checkpoint: the stored readout should say r0=1 r1=0 r2=0 r3=0, with A = 1 and B = 0. (If not, press ↺ reset.) An address is just the register number in binary: 00→r0, 01→r1, 10→r2, 11→r3. read x1 is 00, so port A points at r0 and operand A reads 1.

Click b0 so read x2 becomes 00.

Now both ports point at r0: operand B flips 0 → 1 and the lower wire between the blocks lights as that bit crosses into the ALU. Verify the one thing that matters: the B readout reads 1. The same stored bit is feeding both read outputs in the same instant — no clock, no waiting — which is exactly the trick that lets add grab two operands in a single pass.

Click b0 again to point read x2 back at 01 (r1).

Operand B drops back to 0. That's the add we'll actually run: A = 1 (from r0), B = 0 (from r1).

tell the ALU what to do

The ALU computes ADD, AND, OR, and XOR of A and B simultaneously; the 2-bit op only chooses which one becomes the result: 00→ADD, 01→AND, 10→OR, 11→XOR. Right now op = 00 (ADD) and 1 + 0 = 1, so the result readout is 1.

Before you click: with A = 1 and B = 0, predict what result becomes if op selects AND. Then click op0.

op is now 01 (AND) and result drops 1 → 0 — because 1 AND 0 = 0. The operands never moved; all that changed is which precomputed answer the op-MUX forwards. That's the ALU's whole trick: compute everything, then pick.

Click op0 again to return to op = 00 (ADD).

result goes back to 1. We're set up to execute the real add: operands 1 and 0, operation ADD, answer 1 — now sitting on the wire, waiting.

the anticlimax — a result, but nothing stored

Follow the ALU's output wire: besides leaving on the RIGHT as result, it bends around the bottom of the diagram and arrives back at the register file's write-data input on the left. That returning wire is the feedback loop — the thing that lets a machine build on its own previous answers instead of computing once and stopping.

And yet — look at the stored readout: it still says r3=0. The answer is on the wire but nothing has been remembered. Computing a result and committing it are deliberately separate: a write needs a target (write x3 is already aimed at r3), the enable we = 1, and a clock edge. None of those have happened, so the value sits there, harmless. Next step we make all three line up.

run one full cycle, slowly

Setup confirm: result reads 1, and write x3 is 11 — that's r3 (register 3, the x3 the instruction names), currently 0 (dark), so a change will be impossible to miss.

Click we so it reads we 1.

Writing is now armed — and still nothing has stored. we only unlocks the gate; it's the edge that pushes the value through. (Hover the register file and you'll see the write decoder light up and raise exactly one line — the one aimed at r3.)

Press ↻ run cycle and watch r3.

On the clock edge the result latches into r3: the stored readout flips r3=0 → r3=1 and that register starts glowing. The other three never had write-enable aimed at them, so they hold untouched. You just executed the write-back half of an instruction.

the loop closes — read back what you computed

Here's the payoff of that feedback wire. The 1 you just wrote into r3 is now an ordinary stored bit, indistinguishable from the seeded ones — so a later instruction can read it. Port B is still parked on r1, reading 0; let's aim it at what you just computed.

Click b1 so read x2 becomes 11, pointing port B at r3 — the register you just wrote.

Watch operand B flip 0 → 1: the number you computed last cycle is now an input this cycle, pulled straight back out through the second read port. That round trip — read → operate → write-back → read again — is the entire reason a CPU makes progress. One instruction's output becomes the next instruction's input; strip out the loop and the machine could compute exactly one thing and have nowhere to put it.

why this matters

You're now looking at the literal core of a CPU. Read two registers → run them through the ALU → write the result back is the shape of nearly every arithmetic and logic instruction in the machine; add, sub, and, or, xor, slt all walk this exact path and differ only in the op bits.

Two changes turn this toy into RV32I: widen every wire from 1 bit to 32, and stop driving op/read x1/read x2/write x3 from buttons — instead decode them out of a 32-bit instruction word fetched from memory. Same loop, same blocks, same wires. Everything you built underneath this page existed so that this read → operate → write-back loop could run.
