where we are: the instruction IS the control

On the datapath you set op, raddrA, raddrB, waddr by hand. A CPU sets them from the instruction — and here's the surprise: it barely does any work. A RISC-V instruction carries the register numbers and the operation as plain bit-fields, so "decoding" is mostly slicing the word and wiring each field to the block that uses it — plus one small block of real logic, the control unit.

The 10-bit instruction word on the left splits into five 2-bit fields — opcode (what KIND of instruction: 00 register op, 10 load, 11 store), func (which ALU operation), rs1, rs2, rd. The register and func fields flow straight across to their datapath inputs. Toggle any bit and watch its wire light end to end.

slice the word, name the fields

Set rs1 = 10 (click rs1₁) and rd = 11 (click rd₁ and rd₀).

Follow the lit wires: the rs1 field rides straight out to raddrA (the datapath's first read address), and rd rides out to waddr (the write address). The instruction literally said "read register 2, write register 3," and those numbers are now sitting on the exact wires the register file reads. Nothing computed them — they're just bits of the instruction, relabeled.

the control unit — where the two control bits come from

Click oc₁ to set opcode = 10 (a load), and read the control readout.

The opcode bundle runs into the control block — never out to the datapath. Each row of the block is one line of a truth table: the word matches row 10, so the row lights and raises memToReg — "route memory's data, not the ALU's, into the register write port."

Now click oc₀. opcode = 11 is a store: the 11 row lights instead, raising memWrite — "arm the data memory's write port." Two different consumers, two different lines. An opcode of 01 matches the top row and raises Branch — "this instruction may change where the program goes next." An opcode of 00 matches no row: register ops leave every line at 0.

This is exactly a real CPU's shape: the opcode feeds a combinational control unit — a truth table in gates — whose named outputs (RegWrite, ALUSrc, MemWrite, MemtoReg, Branch…) fan out across the datapath, while func goes separately to the ALU (RISC-V's opcode / funct split). Ours decodes the three lines our CPUs consume: Branch, memToReg, memWrite.

why this matters

This little block is the bridge from "an instruction" to "the datapath running it." Its eight outputs are, top to bottom, the datapath's eight left-edge inputs in the same order — so the next page just drops this decoder beside the datapath and connects them with eight straight wires. Instruction in, control out, datapath obeys.

Widen everything to 32 bits, fix the field positions to the real RV32I layout, and add the bit of control logic this page leaves as bare wire, and you have an actual instruction decoder. What's still missing is where the instruction comes from — the program has to live somewhere and a program counter has to step through it. That's the next thing we build.
