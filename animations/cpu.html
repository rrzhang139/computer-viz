the most basic CPU — a register machine

This is the simplest thing that earns the name CPU: a pure register machine. Its entire repertoire is one move — read two registers, combine them in the ALU, write the answer back to a register. The four registers r0…r3 start preloaded; each instruction names two of them and one operation (ADD / AND / OR / XOR) and stores the result. RISC-V calls these R-type instructions. No data memory, no branches, no jumps — values never leave the register file, and the program runs as one fixed straight line. (Adding memory turns this into the load/store CPU — open the contents menu.)

It's a working datapath, assembled from pages you've already built. Every block is a live door — hover one (PC, instruction memory, decode, register file, ALU) to swap it for an exact copy of the page that builds it; click to drill to the transistors.

There are a lot of wires, so each is colored by what it carries — and every step tells you the one or two colors to watch and to ignore the rest: address, instruction, op, operands A + B, the result, the write-back loop, the clock.

Press ↺ reset. The readout above should now read PC 00 · r0=1 r1=1 r2=0 r3=0. We'll walk one stage at a time, parking the camera on each and pressing ↻ run instruction to advance. fetch: PC → instruction memory

Two colors carry this stage: address (blue) and instr (violet). The PC is a counter holding the address of the current instruction. That address leaves its right edge, wraps down to the instruction memory, and selects one stored word — which comes back on the violet instr bus. No arithmetic here; this stage only locates the next instruction.

Hover the PC and hold still.

The box swaps for its internals: a +1 adder and a register cell, wired in a loop. Right now PC = 00, so the register stores 0 and its two address-out wires sit dark — there's no 1 to light them yet. Beside it the +1 adder glows, already computing 00 + 1 = 01 and holding that next address on its output, waiting for a clock edge.

Now hover instruction memory.

Address 00 lights exactly one of its four stored words — word 0 — and the read MUX forwards that word onto the violet instr bus. The readout names it: INSTR ADD r2,r0,r1. That's the word the rest of the pipeline will act on.

Press ↻ run instruction once and watch the PC.

The clock edge latches the adder's 01 into the register: PC ticks 00 → 01, the low address bit flips 0 → 1, and the register cell lights for the first time. Instruction memory re-reads at the new address — a different word cell lights — and the INSTR readout flips ADD r2,r0,r1 → AND r3,r0,r1. One new word, and everything downstream changes with it.

decode: split the instruction

The fetched word means nothing until it's taken apart. The 10-bit instr bus (/10) enters the decode block, which slices it into five 2-bit fields — an opcode plus four the datapath consumes — sending each out its own bus (/2): rs1 → the register file's read-A address, rs2 → read-B, rd → the write address, and the amber op → the ALU. No clock — the outputs change the instant the input word does.

Hover decode and read the bits.

The word for AND r3,r0,r1 is 00·01·00·01·11 — opcode=00 (a register op), op=01, rs1=00, rs2=01, rd=11. Each bit lands in its own cell, and only the cells holding a 1 light: op lights its low bit, rs2 its low bit, rd lights both, while rs1=00 stays fully dark.

One more thing lives in the decoder: a small control unit that decodes the opcode, truth-table style. Opcode 10 (load) would raise memToReg, 11 (store) would raise memWrite — but every word in this program is opcode 00, so both control lines stay dark, ending at n/c (not connected) dots: this core has no memory to control. The load/store page gives them their consumers.

That bit-pattern is the instruction. A different word would light a different set of cells the same instant it arrived — no clock, no waiting. These four field buses now feed the next two stages. Press next →.

register read: fetch the two operands

Now the addresses become values. Inside the register file, a 2-to-4 address decoder turns each read address into a one-hot select: rs1=00 selects r0, rs2=01 selects r1. The selected cell drives its stored bit into a read MUX, which forwards it out — green A from read port A, teal B from read port B — across the gap toward the ALU. Both reads happen in the same instant, with no clock.

Hover the register file and hold still.

Each of the four cells glows by the bit it stores: r0 and r1 hold 1 (lit), r2 and r3 hold 0 (dark). The two read MUXes are already forwarding r0 and r1, so operands read A=1 and B=1 — confirmed in the readout. The pink rd=11 aims the write port at r3, but nothing writes yet: that needs a clock edge.

Press ↻ run instruction once and watch r3.

The clock edge commits AND's result (1 AND 1 = 1) into r3. Hover the register file again: r3's cell flips dark → lit and the regs readout updates r3=0 → r3=1. The other three cells are untouched — only the one the write port aimed at changed. PC ticks 01 → 10; you've just stored a computed value.

execute: the ALU computes

The two operands cross the central gap into the ALU: green A and teal B. The ALU does not pick one operation and run it — it computes ADD, AND, OR, and XOR of A and B all at once, then an op-MUX forwards just one to the orange result. The amber op bus is the selector: 00→ADD, 01→AND, 10→OR, 11→XOR.

Hover the ALU and hold still.

We're on OR r2,r0,r1 with A=1, B=1. Every operator's output is live at once: AND=1 and OR=1 glow, while XOR=0 and the adder's sum bit stay dark — they computed too, they just produced 0. The op-MUX, told op=10, forwards the OR line, so the result reads 1.

Press ↻ run instruction once and watch the result.

This press stores OR's 1 into r2 (the regs readout ticks r2=0 → r2=1) and advances to the next instruction, XOR r3,r0,r1. Watch the result drop 1 → 0: the operands never moved — both are still 1 — but op is now 11, so the op-MUX stops forwarding OR and forwards the XOR gate instead, and 1 XOR 1 = 0. That's the ALU's whole trick: compute every operation in parallel, then let op pick which precomputed answer survives. PC ticks 10 → 11.

write-back: the result returns to storage

The answer has to get home. In this R-type CPU a result has exactly one source — the ALU — so there's nothing to choose between: the orange result simply turns the corner and runs along the bottom as the pink write-back wire, straight back to the register file's write-data port. That return path is the feedback loop — the reason one instruction's output can become a later one's input. But the value only commits on a clock edge.

Hover the register file and find the write port.

The pink rd address aims the write decoder at one register — here r3 — and the returning result sits on its write-data input. But nothing has changed yet: a write needs the address, the write-enable, and a clock edge to line up at once.

Press ↻ run instruction once and watch r3.

We're on XOR r3,r0,r1, and 1 XOR 1 = 0. On the clock edge that 0 commits to r3: its cell flips lit → dark and the regs readout updates r3=1 → r3=0. PC wraps 11 → 00 and the program loops back to instruction 0. You've watched the whole loop — fetch, decode, read, execute, write-back. (A real CPU adds a second write-back source — data loaded from memory — and a MUX to pick between them; that's the load/store page.)

ground zero — what it does, and what it leaves out

You've watched a complete CPU loop — fetch → decode → execute → write-back — and every block opens into the real circuit underneath, down to transistors. This is the primitive: the smallest machine that already deserves the name "processor." Everything bigger is this same datapath with more sources feeding a few extra MUXes.

It is deliberately stripped down. The caveats, and the pages that will lift each one:

No data memory. It can't load or store — values only live in the four registers, so the ALU result is the only thing that can be written back. That's why there's no write-back MUX here: with a single source, the result wires straight to the register write port. Add a data-memory block and a load instruction and a MUX appears at that port to choose ALU-result vs. loaded-data — the classic memory stage. Open the load/store page to watch exactly that.

No control flow. The PC just counts up, so the program is a straight line — no branch, no jump, no loops. Add a PC-select MUX and a comparison in front of the counter and you get if and while.

Toy width. 1-bit data, 4 registers, 4 operations. Widen the wires to 32 bits, the file to 32 registers, and the op set to full RV32I — the shape doesn't change — and it runs real programs.

No overlap. This single-cycle view runs one instruction at a time. A pipeline inserts registers between the stages so several instructions are in flight at once — the timing layer built on top of exactly this datapath.

So treat this as the trunk. The memory page, the branch page, and the pipeline page all graft onto it — same blocks, more doors.
