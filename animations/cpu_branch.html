the branch CPU — choosing the next instruction

Until now the PC had exactly one future: PC + 1. This page gives it a second one. Two new pieces do it: a branch target wire that loops the instruction's target field back into fetch, and a branch taken? block beside the ALU that decides — taken = branch and the two registers are equal.

Hover the PC: its +1 loop has been opened by a MUX. Input 0 is the sequential future (+1), input 1 is the branch target, and taken selects. That MUX is the whole lesson — the load/store page promised it: branches add a PC-source MUX.

The new instruction is BEQ rs1,rs2,→t: if rs1 = rs2, the next PC is t; otherwise fall through to PC+1. Press ↺ reset: PC 00 · r0=1 r1=1 r2=0 r3=0. Tip: click any paragraph to spotlight what it names (click again or Esc clears).

warm-up — one sequential step

First, feel the default. PC 00 fetches AND r3,r0,r1 — an ordinary R-type from the last two pages. No branch anywhere: the taken wire stays dark, the PC MUX picks +1.

Press ↻ run instruction: r3 becomes 1 and PC ticks 00 → 01. Sequential life: each cell of instruction memory in order, one per clock.

the branch — trace the decision

Now PC 01 holds BEQ r0,r1,→3: if r0 = r1, jump to cell 3. Click each numbered step to light its path.

1 · fetch. The PC (01) addresses instruction memory; the BEQ word rides out on the violet instr bus.

2 · decode. opcode=01 — the branch row. The control unit raises the amber branch line, which runs to the branch taken? block. The ALU op is set to XOR: the comparison is free — two values are equal exactly when their XOR is 0.

3 · the target loops back. The instruction's target field (3) runs all the way back LEFT into the PC's MUX — the blue target pair. Execute is reaching back into fetch: that backwards wire is what makes a branch a branch.

4 · compare. The register file reads r0=1 and r1=1; the ALU XORs them → 0. Equal.

5 · decide. branch=1 and equal → taken=1. The decision rides the long top wire back to the PC MUX select. Click next → to pulse the clock.

the branch — run it and watch the PC jump

Everything is staged: the MUX is holding the target 3, taken=1 selects it.

Press ↻ run instruction and watch the PC readout.

PC jumps 01 → 11 — not 10. No register changed, no memory changed: a branch's entire effect is where the machine looks next.

the skipped cell — the key idea

Hover instruction memory. The fetch lit cell M3 — and cell M2, outlined red, was skipped. It sits between where you were and where you are, and the machine simply never looked at it.

The proof is in the registers: M2 holds ADD r2,r0,r1, which would have set r2=1. Check the readout — r2 is still 0. An instruction that is never fetched never happened.

And the counterfactual: had r0 ≠ r1, taken would stay 0, the MUX would pick +1, and M2 would have run like any other cell. Same wires, two futures — the data decided.

land at the target

The machine carries on from cell 3 as if nothing happened — a branch is not an exception, just a different next address. M3 holds XOR r3,r0,r1.

Press ↻ run instruction: r3 flips 1 → 0 (1 XOR 1). The target instruction executed exactly like a sequential one — and the PC resumes counting from 11.

why this matters — control flow is born

Every if, every while, every function call in every program compiles down to exactly this: a compare, a taken decision, a PC-source MUX. Point the target backwards and you have a loop — the same four memory cells can now run forever.

The MUX is the whole idea, again. Load/store added a write-back MUX to choose a register's source; the branch adds a PC MUX to choose the machine's source. New capability = new source + a MUX. Same move every time.

Toy → real. RV32I's beq computes its target as PC + offset with a dedicated adder (here the target rides in the instruction directly), the compare is a full 32-bit subtract-and-test-zero, and the decoder's truth table grows its Branch row exactly like this one. And the fact that the machine has already fetched PC+1 before the decision arrives is the control hazard that makes pipelines interesting — the next page's problem.
