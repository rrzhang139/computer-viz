the branch CPU — choosing the next instruction

Until now the PC had exactly one possible future: PC + 1. This page gives it a second one. Two new pieces make that possible: a branch target wire that carries the instruction's target address back into fetch, and a PCSrc block beside the ALU that makes the decision. These are the standard names from computer-architecture courses: the decoder raises Branch, the ALU reports Zero (the two values were equal), and PCSrc — Branch AND Zero — tells the PC which future to take.

Hover the PC and look inside. Its +1 loop now runs through a MUX — a switch with several inputs, one output, and a select wire that decides which input passes through. Input 0 of this MUX is the incremented PC (the sequential future); input 1 is the branch target (the other future); the select wire is PCSrc. The load/store page promised this: adding a capability means adding a source and a MUX to choose it.

The new instruction is BEQ rs1,rs2,→t — "branch if equal": if register rs1 equals register rs2, the next instruction comes from address t; otherwise the program continues to PC+1 as usual. Press ↺ reset to start clean: PC 00, registers r0=1 r1=1 r2=0 r3=0. Tip: click any paragraph to spotlight the wires and blocks it names (click again or Esc to clear).

warm-up — one ordinary sequential step

Before the branch, feel the default. PC 00 fetches AND r3,r0,r1 — an ordinary register instruction from the previous pages. Nothing branch-related is active: the Branch line stays dark, so PCSrc stays 0 and the PC's MUX passes the incremented value through.

Press ↻ run instruction. Register r3 becomes 1, and the PC steps from 00 to 01. This is sequential life: the machine walks through instruction memory one cell per clock tick.

the branch — trace the decision

Now PC 01 holds BEQ r0,r1,→3: if register r0 equals register r1, jump to cell 3. Click each numbered paragraph to light up that part of the path.

1 · fetch. The PC (01) addresses instruction memory, and the BEQ word rides out on the violet instr bus — exactly like any other instruction so far.

2 · decode. The opcode is 01 — the branch opcode. Hover decode: inside its control unit, the 01 · beq row of the truth table lights up and raises the amber Branch line. The control unit also forces the ALU's operation to XOR — the comparison — no matter what the func field says. (This is the textbook ALUOp trick: a branch hijacks the ALU to do its subtraction, because two values are equal exactly when their difference is zero, and one-bit subtraction is XOR.)

3 · the target travels to the PC. The instruction's last field carries the target address, 3. Watch the pink wires: the same field that normally names the destination register (rd, pink, decode → register file) is reused by a branch as the target, and a pink pair carries it from the decoder all the way left into the PC's MUX. This wire is permanently connected — it always carries whatever is in that field. It causes nothing by itself; the MUX simply ignores it unless the select line says otherwise.

4 · compare. The register file reads r0 = 1 and r1 = 1, and the ALU computes their XOR, which is 0. A result of zero means the two values were equal.

5 · decide. Hover the PCSrc block: the ALU's zero result and the raised Branch line meet at an AND gate, so PCSrc goes to 1. That decision rides the long wire across the top of the datapath back to the PC's MUX select. Everything is now staged — click next → to continue.

the branch — run it and watch the jump

The target 3 is waiting at the PC MUX's input 1, and PCSrc = 1 has switched the MUX to pass it through. The next clock edge loads it into the PC.

Press ↻ run instruction, then hover instruction memory and watch which cell is lit. The fetch was on the second cell (M1); now it is on the fourth (M3). It never touched M2.

The PC readout jumped from 01 to 11 — not 10. No register changed and no memory changed: a branch's entire effect is where the machine looks next.

the skipped cell — the key idea

Hover instruction memory. The fetch lit cell M3 — and cell M2, outlined in red, was skipped. It sits between where the program was and where it is now, and the machine simply never looked at it.

The registers prove it. M2 holds ADD r2,r0,r1, which would have set r2 to 1. Check the readout: r2 is still 0. An instruction that is never fetched never happened.

And the other future was one wire away: had r0 and r1 differed, the ALU's result would not have been zero, PCSrc would have stayed 0, the MUX would have passed PC+1 through, and M2 would have run like any other cell. Same wires either way — the data decided.

land at the target

The machine carries on from cell 3 as if nothing unusual happened — a branch is not an exception, just a different next address. M3 holds XOR r3,r0,r1.

Press ↻ run instruction. r3 flips from 1 to 0 (1 XOR 1 = 0) — the target instruction executed exactly like a sequential one, and the PC resumes counting from 11.

why this matters — control flow is born

Every if, every while, every function call in every program compiles down to exactly this: a comparison, a PCSrc decision, and a PC MUX. Point the target backwards and you have a loop — the same four memory cells can now run forever.

The MUX is the whole idea, again. Load/store added a write-back MUX so a register could choose its source; the branch adds a PC MUX so the machine can choose its next instruction. New capability, new source, new MUX — the same move every time.

Toy → real. A real RV32I beq computes its target as PC + offset with a dedicated adder (ours carries the target inside the instruction), the comparison is a full 32-bit subtraction whose Zero flag comes straight off the ALU, and the control unit's truth table grows a Branch row exactly like ours. One more thing to notice: the decision needed the fetch, the decode, the register read, and the ALU — the longest path on the page — before fetch knew where to look next. Pipelined CPUs refuse to wait and guess instead. That problem is called the control hazard, and it is the next page's story.
