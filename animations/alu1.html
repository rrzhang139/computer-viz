where we are: the ALU, one bit wide

The ALU is the CPU's calculator: hand it two operands and a 2-bit op and it adds, ANDs, ORs, or XORs them. We build it one bit wide — the true design unit — because a wider ALU is nothing more than this exact slice replicated, one copy per bit.

A slice takes one bit of each operand (A, B), a carry-in Cin, and the 2-bit op, and produces one result bit Y plus a carry-out Cout. Inputs on the LEFT, results on the RIGHT. Press ↺ reset so every input starts at 0, and watch the readout strip — it shows A, B, Cin, op, Y, and Cout live. This is the same "build the 1-bit cell, then replicate" move you saw take the full adder up to the 4-bit adder.

four tiny machines, computed at once

The four blocks on the left all run together on the same A and B, every cycle: the full adder computes A + B + Cin (one sum bit plus a carry), and the single AND, OR, and XOR gates each compute their one-bit result. Toggle A and B and you'll see several blocks light at once — four answers sitting ready before op has chosen anything.

Click the full adder to drill into /fulladder.html — it's literally the full adder you built earlier (two half-adders and an OR), now serving as the ADD path inside the ALU.

op picks exactly one answer

The 2-bit op drives a 4-to-1 MUX on the right: 00 → A+B, 01 → AND, 10 → OR, 11 → XOR. Whichever op points at flows out to Y; the other three results are computed and discarded. Click the MUX to drill into /mux.html — the exact primitive doing the selecting.

Set both operands high: A = 1, B = 1. Leave op = 00 (ADD) and read the result in the next step — first just confirm all the operand wires into the blocks are lit.

same A and B, four different Y

With A = 1 and B = 1 still loaded (and Cin = 0), step op through every value and read Y:

op=00 ADD → Y = 0 with Cout = 1, because 1 + 1 = 10: the sum bit is 0 and the carry escapes (more on that next). Flip op0 → op=01 AND → Y = 1 (both bits are 1), and the carry is now irrelevant. op=10 OR → Y = 1. op=11 XOR → Y = 0 (the bits are equal, so they differ in zero places). The operands never moved; only the select did.

the carry is the one wire between slices

Only the ADD path uses Cin and Cout — the bitwise ops compute a carry internally too, but the MUX never forwards it, so it stays harmless. See it move the result: with op = 00, set A = 1, B = 0, Cin = 0 → Y = 1, Cout = 0. Now turn Cin = 1 → the slice is adding three 1-bit numbers (1+0+1 = 10), so Y = 0 and Cout = 1.

That Cout is exactly what feeds the next slice's Cin when you stack these into a wider ALU. Note the division of labor: control (op) is broadcast identically to every slice, data (A, B) is private to each slice, and the carry is the only wire that runs from one slice to the next.

how four of these become a 4-bit add

Make the chaining concrete. Imagine four of these slices stacked, computing 0001 + 0001. The bit-0 slice gets A=1, B=1, Cin=0 → Y=0, Cout=1 (you just saw this). That Cout=1 becomes the bit-1 slice's Cin, where A=0, B=0, Cin=1 → Y=1, Cout=0. The upper two slices add 0+0+0 → 0.

Read the result bits top-to-bottom: 0010 = 2. The carry rippled exactly one slice up and stopped. Nothing here is special-cased per position — every slice is this same page, and the Cout → Cin handoff is the whole mechanism that turns four one-bit adds into one four-bit add.

why this matters

This one slice is the true atom of an ALU. Stack 4 of them — broadcasting op to all, chaining Cout → Cin up the column — and you have a 4-bit ALU; stack 32 and you have the RV32I ALU that runs add x3, x1, x2. The datapath page embeds exactly this slice as its ALU.

Widening a datapath is never "redesign the logic" — it's "replicate the bit-slice and broadcast the control." The same move widened the adder and the register file. The slice is the unit; the width is just how many you set side by side.
