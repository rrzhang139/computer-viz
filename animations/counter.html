where we are: the machine knows where it is

You have storage (the register) and arithmetic (the adder). Wire the register's output back through the adder into its own input, and you've crossed a threshold: a circuit that updates itself on every clock edge.

This is a program counter. It holds an address. Every clock tick, it advances. That's how a CPU knows which instruction to fetch next — without a PC, the machine has no sense of "now I am here".

the layout

On the left, the 4-bit register from the previous page. On the right, the 4-bit adder, with its B input hardwired to 0001 — so its job is simply "output A + 1".

The register's outputs Q3..Q0 feed straight into the adder's A3..A0. The adder's outputs S3..S0 loop back along the bottom into the register's D3..D0. That loop is the whole trick.

press pulse — PC increments

Hit pulse CLK 0→1→0. Watch carefully: the rising edge transfers the adder's output into the register. The register's Q now equals the OLD adder output — which was OLD-PC + 1. So PC has advanced by one.

Immediately after the edge, the adder sees the new Q and recomputes — so "next" already shows PC + 1 again, queued up for the next pulse. Keep pulsing: 0 → 1 → 2 → 3 → ... → 15 → wraps to 0.

why doesn't it race?

The feedback loop is "obviously" a problem: Q feeds A, A drives S, S feeds back to D, and D would seem to feed Q. If the register were transparent, this would oscillate uncontrollably as the adder kept incrementing its own output.

It doesn't because the register is built from edge-triggered DFFs. Between clock edges, D is ignored — the loop is open. Only at the brief rising edge does D snap into Q, and by then the new adder output is "the value to capture next time," not "the value to chase right now."

This is exactly why we spent so long on edge-triggering. A latch-based register here would explode.

combinational vs sequential, made physical

Look at the diagram. The adder (right) is purely combinational: between clock pulses, its output is whatever Q + 1 happens to be, immediately. The register (left) is purely sequential: it only changes at clock edges.

Combinational logic computes what could be next. Sequential logic decides when next happens. Every interesting circuit in a CPU has this same shape — a hot, always-computing combinational cloud feeding a clock-gated state element.

where the constants come from

Hover the adder box. You'll see each FA's left side has THREE inputs: A (from the register's Q wire, lights up per-bit), B (a short stub labeled "B=1 (Vdd)" or "B=0 (GND)"), and Cin (the carry from the FA below). Where do the B and Cin values physically come from?

Not from anywhere — they're hardwired. The B bit of FA0 is just a wire permanently soldered to the chip's Vdd rail (for B=1) or GND rail (for B=0). FA0's Cin is similarly tied to GND because we're only doing addition (subtraction would tie it to Vdd and flip the B inputs through inverters). Same with FA1, FA2, FA3 — each B is either a tie to Vdd or to GND, fixed forever at chip-design time.

This is what the step: +1 / step: +4 toggle is really showing you: two different physical circuits, not one circuit that switches at runtime. A real chip can't dynamically rewire which rail B is tied to — that would require multiplexers (which IS what branch/jump logic uses to feed a different value into the PC's adder, but that's a different B source, not a runtime tie-change).

flip to +4 — the CPU's actual fetch stride

Hit step: +1 to toggle into step: +4 mode. The adder's hardwired B input switches from 0001 to 0100 — a constant 4 instead of a constant 1. The "+1 feedback loop" label updates to "+4 feedback loop"; the adder relabels itself. Nothing else changes in the topology — the register, the feedback wires, the clock fan-out are all identical. Only the constant being added is different.

Now pulse the clock and watch PC step through 0 → 4 → 8 → 12 → 0 (wraps because 16 = 10000 overflows the 4-bit width). At each pulse, the same single carry rises through the same adder lanes — but starting at bit 2 instead of bit 0, because the "+1" lives in bit 2 of the B constant.

Open the adder's hover overlay during a pulse to watch the lit B-constant indicator shift from FA0 (in +1 mode) to FA2 (in +4 mode). That single bit's position is the only physical difference between a demo counter and a real CPU's program counter.

why this matters

The CPU's instruction-fetch loop is exactly this circuit, scaled up. RV32I has a 32-bit PC; instructions are 4 bytes long; so the adder adds +4 on every fetch (which you just simulated above). Branches and jumps inject a different value into the adder, replacing the constant — same loop, different addend.

Now you can see the shape of a CPU. The PC chases itself through memory, the fetched instruction routes data through an ALU (a generalized adder), and the ALU's output lands back in a register on the next clock — same loop, more cargo. We've just built the spine.
