the load/store CPU — a round trip through memory

This is the R-type core with two additions: a data memory beside the register file and a revived write-back MUX. Together they unlock the two instructions every program leans on — SW (store: Mem[rs1] = rs2) and LW (load: rd = Mem[rs1]).

We'll trace one store end-to-end, run it, then trace one load that reads the value right back. Two new colors to watch: the light-blue loaded-data wire out of memory, and the amber control lines memWrite / memToReg — born inside the decoder, where a small control unit decodes the instruction's opcode and raises the matching line.

Press ↺ reset: the readout should read PC 00 · r0=1 r1=1 r2=0 r3=0 · M0=0 M1=0 (memory empty). Tip: click any paragraph to spotlight the wires it names and open the blocks it describes (click again or Esc clears).

store — trace the whole path (before running)

A store runs data out of the register file into memory. This instruction is SW r1,(r0): write r1's value into the memory cell addressed by r0. Click each numbered step to light its part of the path — nothing commits until you run it.

1 · fetch. The PC (00) addresses instruction memory, which hands back the SW word on the violet instr bus.

2 · decode. decode splits the word: opcode=11 (a store), rs1=r0 (the address), rs2=r1 (the data). Its control unit matches the 11 · store row of its truth table and raises the amber memWrite line, which runs down to the data memory's write-enable — hover decode to see the row lit.

3 · register read. The register file reads both ports: r0=1 heads for the address, r1=1 is the value to store.

4 · address. The ALU passes r0 straight through as the effective address — which memory cell to write (cell 1).

5 · data memory. The write port now has all three: address = 1, write-data = r1=1, memWrite=1. It's armed — but a write only commits on a clock edge. Click next → to pulse it.

store — run it

Everything was staged in the last step. Now we pulse the clock and the write commits.

Press ↻ run instruction once and watch cell M1.

On the clock edge the write port stores 1 into M1 — the cell at address 1 (cells count up from M0): the M1 readout flips 0 → 1 (hover the data memory — that cell now glows). No register changed: a store touches memory only. PC ticks 00 → 01.

load — trace the whole path (before running)

A load is the reverse trip: read a memory cell back into a register. This instruction is LW r2,(r0): read the cell addressed by r0 into r2 — and that cell holds the 1 you just stored.

1 · fetch. The PC (01) fetches the LW word on the violet instr bus.

2 · decode. opcode=10 (a load): rs1=r0 (the address), rd=r2 (the destination). This time the control unit matches its 10 · load row and raises memToReg: the write-back must come from memory, not the ALU.

3 · read + address. r0=1 is read, and the ALU passes it as the address — cell 1.

4 · data memory read. Address 1 selects cell M1; its stored 1 rides out on the light-blue loaded-data wire and climbs up to the write-back MUX beside the ALU.

5 · write-back MUX. With memToReg=1 the MUX takes its in1 (loaded data) instead of in0 (the ALU result), and routes it to r2's write port. Click next → to commit it.

load — run it

The loaded value is staged at the MUX, routed to r2. The clock edge latches it.

Press ↻ run instruction once and watch r2.

The clock writes the loaded 1 into r2: the regs readout ticks r2=0 → r2=1. The round trip is complete — you stored a value to memory, then read it straight back into a register. PC ticks 01 → 10.

why this matters — the memory stage

You've added the classic memory stage: fetch → decode → execute → MEM → write-back. Two pieces did it — a data memory with a write port, and a write-back MUX so a register can be fed by either the ALU or memory.

The MUX is the whole idea. Adding a capability to a CPU usually means adding a source and a MUX to choose it: branches add a PC-source MUX, immediates add an ALU-input MUX. Same move every time.

Toy → real. Widen the data to 32 bits, give memory real address bits, compute the address as base + offset in the ALU (here it just passes rs1), and grow the decoder's control-unit truth table from two rows to one per opcode — RegWrite, ALUSrc, Branch join memWrite and memToReg. The store/load/MUX shape is exactly what RV32I runs.
