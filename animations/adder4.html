what you're looking at

Four full adders chained together — FA0 on the right (the least-significant bit), FA3 on the left (the most-significant). Together they add two 4-bit binary numbers.

Each FA_i takes three inputs (Ai, Bi, and a carry from the bit below it) and produces two outputs (Si and a carry to the bit above). The carries flow right to left through the wires labeled C01, C12, C23. This is called a ripple-carry adder.

Hover any FA to see its half-adder + half-adder + OR interior reflecting that bit's current state.

0 + 0 = 0

Hit ↺ reset. All inputs are 0. Everything is dark. Display reads 0 + 0 = 00000.

No FA needs to do anything: each one's three inputs are 0, so each output is 0. No carry is generated anywhere.

1 + 1 = 0010 — the smallest carry

Click A0 and B0. Now A = 0001, B = 0001.

FA0 sees A0=1, B0=1, Cin=0. That's the case we already know: sum=0, but a carry is generated. C01 lights up.

FA1 receives that carry. It now has A1=0, B1=0, but Cin=1. Exactly one input is 1, so FA1's S=1. No further carry. The result: 0010 = 2.

FA2 and FA3 stay dark — the carry never reached them.

5 + 3 = 1000 — multi-stage ripple

Set A = 0101 (click A2 and A0) and B = 0011 (click B1 in addition to B0). Now A = 0101 (decimal 5) and B = 0011 (decimal 3).

FA0: 1+1+0 = 10. S0=0, carry generated. C01 fires.
FA1: 0+1+1 = 10. S1=0, another carry. C12 fires.
FA2: 1+0+1 = 10. S2=0, another carry. C23 fires.
FA3: 0+0+1 = 1. S3=1, no carry out.

Result: 1000 = 8. Three carries in a row, all glowing across the bottom of the canvas. This is what "ripple" means — the carry travels from FA0 all the way up to FA3.

15 + 1 = 10000 — max overflow

Set A = 1111 (all four A bits) and B = 0001. A is decimal 15, the maximum 4-bit value.

Adding just 1 forces a carry at every position: 1+1+0=10 at FA0, then 1+0+1=10 at FA1, 1+0+1 at FA2, 1+0+1 at FA3. Every carry wire lights up — Cin (wait, Cin is 0), C01, C12, C23, and finally Cout = 1.

The result is 5 bits: 10000 = 16. We've overflowed the 4-bit range. Cout is how a 4-bit adder tells you the sum didn't fit.

the ripple delay

Notice that FA3's output depends on a carry from FA2, which depends on FA1's carry, which depends on FA0's. The bits are computed in sequence, not in parallel.

In real silicon each FA takes ~50-100 picoseconds to settle. For a 4-bit adder, the worst-case delay is 4× that (the carry has to propagate through all four). For a 64-bit adder it would be 64×. That's why modern CPUs use carry-lookahead or carry-save adders that compute carries faster, in parallel.

But the underlying logic is exactly what you see here. The optimizations are about when each carry is computed; what's computed is the same.

why this matters

This is integer addition. Every add instruction in every CPU ultimately runs a circuit shaped like this — typically 32 or 64 bits wide. Subtract reuses the same circuit with B inverted plus Cin=1 (two's complement). Multiply is repeated addition. Comparisons are subtraction with the carry inspected.

Combined with the register file from before (storage) and a clock (timing), the 4-bit adder is enough to build a complete, programmable arithmetic machine. Stack everything we've built on this page and you have a CPU.
