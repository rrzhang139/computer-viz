where we are: pick one of many

The decoder on the last page turned a number into a single chosen wire. This page asks the natural follow-up: now that you can point at one of several things, how do you actually fetch the value sitting there? That's the multiplexer — a switchboard that takes N data inputs and a binary select, and routes exactly the selected input through to one output.

Toggle the select bits s1 and s0 and watch which input wire connects to out. A MUX is the dual of the decoder: the decoder takes an address and lights one of N wires; the MUX takes one of N wires (lit by a decoder!) and uses it to wave a single data value through. Same idea, run backwards.

the structure: a decoder, four ANDs, one OR

Open the hood and the MUX is built entirely from primitives you already have. The block in the upper-center is a literal 2-to-4 decoder — the very circuit from the previous page (click it to drill into /decoder.html). It chews on (s1, s0) and raises one of four one-hot lines.

Each one-hot line is one input of an AND gate; the AND's other input is one of the four data bits. An AND only passes a 1 when both inputs are high — so a data bit can only get through the AND whose decoder line is lit. The other three ANDs see a 0 from their (dark) decoder line and are clamped to 0, ignoring their data entirely. The one-hot acts as a gate: exactly one door is unlocked.

Finally all four AND outputs pour into one 4-input OR. Since three of them are guaranteed 0, the OR just passes the one survivor through. out equals whichever input the select pointed at.

walk all four selects

Set all four inputs to 1 so any selection should yield a 1. Now cycle s1, s0 through 00, 01, 10, 11 and watch the lit AND slide down the column — the open door moves, but because every input is 1, out stays high the whole time.

Now flip in2 to 0 and select 10 (s1=1, s0=0). out drops to 0 — the addressed input genuinely holds a 0, and the MUX faithfully forwards it. Select anything else and out springs back up. The MUX always shows you the value living at the addressed input, nothing more.

the same decoder, used in reverse

Notice what just happened: you've now seen the decoder play both roles. On /decoder.html it stood alone — address in, one-hot out, a routing curiosity. Here it sits embedded, producing the identical one-hot, but the MUX bends that one-hot to a new purpose: gating data through.

Decoder: "I have an address — light up one of N wires."
MUX: "I have N values and need one — use the decoder's one-hot as the gate." Same primitive, opposite direction. This pairing is so fundamental that you'll find it inside almost everything above this page.

why this matters

A CPU is, to a first approximation, a pile of MUXes steering data between a few busy blocks. The register file's read port is a 32-to-1 MUX picking one of 32 stored values. The ALU's result selector is a MUX choosing add vs. and vs. or by opcode — you'll build that exact one on the ALU page. The next-PC selector is a MUX choosing sequential vs. branch vs. jump. Even "which word do I fetch from memory?" is a giant MUX dressed up in addressing logic.

You've built the 4-input version from a decoder, some ANDs, and an OR. Widen the data to 32 bits and the select to 5 bits and this is the read port of an RV32I register file — which is exactly where you'll see it embedded next.
