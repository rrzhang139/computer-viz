where we are: where the convergence happens

So far each page added one idea. This page adds none — it's a reunion. A register file is a small bank of registers the CPU names by number, and to pull that off it leans on three primitives you already built: the decoder that turns an address into a one-hot wire, the DFF cells that actually remember the bits, and the MUX that picks one stored bit back out.

This one is deliberately tiny — 4 registers of one bit each — so every wire is a single bit and the counts stay honest (a real register just stacks 32 of these bit-slices). But it is exactly where x1, x2, x3 live in the demo program. Over the next four steps you'll do the round trip every instruction's result takes: store a bit into one register, prove the other three never budged, then read it back — on two ports at once. Press next → to start.

aim the write: turn a number into a target

Mission: store a 1 into reg2, which is dark right now. (Check the stored readout above the canvas — it should say r0=0 r1=0 r2=0 r3=1; if not, press ↺ reset.) Writing is the dangerous half of a register file — the failure mode is hitting the wrong register — so the first job is aiming.

Click we so it reads we 1.

The decoder wakes up and exactly one line leaves it — wen0, running across to reg0's AND gate. Why reg0? The write address is still 00, and an address is just the register's number in binary: 00→reg0, 01→reg1, 10→reg2, 11→reg3. We're aimed at the wrong register.

Click waddr1 — but first, guess which AND gate the bright line will jump to.

The address became 10 — binary for 2 — and the lit line jumped to reg2's gate. That's the decoder's whole job: turn a number into one raised wire. And the three dark wen lines are a promise: those three registers cannot change this cycle, no matter what.

Hover the decoder and hold still — the box swaps for its real internals, lit with this exact address.

Inside, two inverters make the address bits' opposites (ā1, ā0), and each of the four AND gates listens for one exact combination. With 10 on the wires, only the gate watching for a1·ā0 finds all its inputs high — and its output is the lit wen2 line you just watched jump. Click to drill in with this address loaded.

fire the edge: data + clock = a stored bit

Click wdata so it reads wdata 1.

The wire from the bottom-left lights up, runs up the bus, and fans into all four registers' D inputs — every register is shown the new bit. And yet nothing happens: the readout still says r2=0. Data alone never writes. A register only looks at D on a clock edge, and no edge has arrived.

Press ↻ clock write and follow the pulse.

The edge races down the shared clk line into all four AND gates at once — but only reg2's gate has its wen raised, so only the wire dropping into reg2's clock fires. reg2 latches the 1 and starts glowing, the readout flips to r2=1, and the other three never saw an edge at all. The decoder chose, the AND gate delivered the edge, the register remembered.

Hover reg2 to see where the bit actually lives.

The preview is the real D flip-flop: your wdata arriving on D, the gated clock landing on CLK, and the two latches inside — the master that sampled the 1, the slave now publishing it on Q — both glowing. A one-bit register is a DFF. Click to drill in.

try to wreck it: the master safety switch

A CPU clocks its register file on every cycle of every program, so what keeps your bit alive for the thousands of cycles between writes is everything that doesn't happen. Let's attack the bit we just stored.

Click we back to we 0, click wdata back to wdata 0, then press ↻ clock write again.

The pulse runs exactly as before — and nothing changes. r2 is still 1. With we = 0 the decoder raises no wen line at all, so all four AND gates swallow the edge and not one gated clock fires. we (write-enable) is the master "may anything change this cycle?" switch: flip it off and the file is untouchable, no matter what the clock or the data wires are doing.

read it back — on two ports at once

Reading is the easy half: no clock, no enable, no danger. Each read port is the 4-to-1 MUX you already met, reaching into the same four stored bits, addressed by the same binary-number rule.

Click raddrA1 so read A's address is 10.

rdata flips from 0 to 1 the instant the wires settle — that's the bit you stored two steps ago, pulled back out through MUX A. No clock edge needed: reading is purely combinational, always live.

Now click raddrB1 so read B's address is 11.

rdataB shows 1 — reg3's bit — while port A keeps reading reg2. Two different registers, read at the same instant, neither port disturbing storage. That's why there are two ports: an instruction like add x3, x1, x2 needs x1 and x2 in the same cycle.

Hover MUX B and you'll find an old friend inside.

Another decoder — the same trick as the write side, now aimed at reading. It takes raddrB = 11 and raises its fourth select line, which opens the fourth AND gate — the one sitting on reg3's wire. The stored 1 rides through that open gate, the OR funnels it out, and that is rdataB. A read is just: decode the address, open one gate, let the stored bit through. Click either MUX to drill in with these four bits and that port's address loaded.

Write on a clock edge, read whenever you like — that split is the entire contract of a register file, and you've now done both ends of it by hand.

why this matters

Every RISC-V instruction names registers. add x3, x1, x2 says "read x1, read x2, write x3" — that's exactly the two read ports and one write port you see here, each addressed by a register number. The real one just isn't shy about size: 32 entries of 32 bits instead of 4 entries of 1 bit.

Widen each register to 32 bits and grow the addresses to 5 bits (so the write side becomes a 5-to-32 decoder and each read port a 32-to-1 MUX) and you are looking at the literal heart of the CPU's datapath. The next page pairs this with the ALU — rdata and rdataB become the ALU's two operands — and the values stored here finally get something done to them.
