where we are: storage you address by number

A register file is a tiny bank of registers the CPU reads and writes by address. This one holds 4 one-bit registers. It's where x1, x2, x3 live in the demo program — give an address, get (or set) that register's value.

It is the convergence of three primitives you already built: a decoder for the write port, DFF cells for storage, and a MUX for the read port.

the read port is a MUX

Set raddr1, raddr0 and watch rdata on the right. The read address drives the 4-to-1 MUX's select, so the MUX routes the addressed register's Q straight out. Reading is combinational — no clock needed; rdata always shows the current value of the addressed register.

Click the MUX to drill into /mux.html with the four register values and the read address already loaded.

the write port is a decoder

The 2-to-4 decoder turns the write address into a one-hot line wen0..wen3 — exactly one is high, and only when we = 1. Set we and a write address and watch one decoder output light up. Click the decoder to drill into /decoder.html.

This is the decoder's other role: standalone it's "address → one-hot"; here that one-hot becomes "which register may be written".

writing: the one-hot gates the clock

Each wen line is ANDed with clk (the four & gates), and that gated clock is the register's clock. So when you press ↻ clock write, only the addressed register sees a rising edge and latches wdata; the other three hold their value.

Try it: set we=1, waddr=10, wdata=1, press clock write. Only reg2 updates. Click any register to drill into its /dff.html.

why this matters

Every RISC-V instruction names registers: add x3, x1, x2 reads x1 and x2 and writes x3. That's two read ports and one write port on a file like this — just 32 entries of 32 bits instead of 4 entries of 1 bit.

Widen the cells to 32 DFFs, the addresses to 5 bits (a 5-to-32 decoder, a 32-to-1 MUX per read port), add a second read port, and you have the heart of the CPU's datapath.
