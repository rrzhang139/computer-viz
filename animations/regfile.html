where we are: where the convergence happens

So far each page added one idea. This page adds none — it's a reunion. A register file is a small bank of registers the CPU names by number, and to pull that off it leans on three primitives you already built: the decoder that turns an address into a one-hot wire, the DFF cells that actually remember the bits, and the MUX that picks one stored bit back out.

This one is deliberately tiny — 4 registers of one bit each — so every wire is a single bit and the counts stay honest (a real register just stacks 32 of these bit-slices). But it is exactly where x1, x2, x3 live in the demo program. Hand it an address, get that register's value; hand it an address plus data plus a clock edge, set it. Below this is silicon; above it is the datapath that runs add x3, x1, x2.

reading: a MUX you already met

Start on the right, the easy half. A register glows when it's holding a 1 — so with the preloaded pattern (reg0=1, reg1=0, reg2=0, reg3=1) you'll see reg0 and reg3 lit and reg1, reg2 dark. The 2-bit raddr is simply the register's number written in binary: 00 → reg0, 01 → reg1, 10 → reg2, 11 → reg3 (read the two bits as a number, that's the register it picks). Toggle raddr1/raddr0 and watch rdata become whatever bit the addressed register is glowing (or not). Those two bits drive the select of a 4-to-1 MUX — the very same one from the MUX page — which reaches into the four registers' stored bits and forwards just the addressed one.

There are two read ports: MUX A (raddr → rdata) and MUX B (raddrB → rdataB). Both reach into the same four registers; each picks independently. That's because one instruction reads two registers at once — add x3, x1, x2 needs x1 on one port and x2 on the other. Toggle read B and watch rdataB change while rdata stays put. Reading is purely combinational — no clock; change an address and its output updates the instant the wires settle. Click either MUX to drill into /mux.html with these four values and that port's address loaded.

aiming the write: the decoder picks the target

Writing is harder — the danger is hitting the wrong register. The 2-to-4 decoder raises exactly one of wen0..wen3, and which one is the same binary-number rule as reading: 00 → reg0, 01 → reg1, 10 → reg2, 11 → reg3 — the write address is just the register's number.

Watch it: set we = 1 and the decoder lights; set waddr = 01 (waddr0 on) and one line — wen1 — turns on and runs across to reg1's AND gate while the other three stay dark; toggle to 11 and the bright line jumps to reg3's gate. That lit line answers a sharper question than it did on the decoder page — which register is even allowed to change — and the three dark lines are the three kept safe. Click the decoder to drill into /decoder.html.

writing: the one-hot gates the clock

Now the punchline that ties the decoder to the registers. Each wen line is ANDed with the clock (the four AND gates in the middle), and that gated signal — not the raw clock — becomes each register's clock. So a rising edge only reaches the one register whose wen is high. The other three never see an edge, so their registers hold their value, exactly as a register does when its clock stays quiet.

Now actually store something — go slowly and watch it happen. Set we = 1 and waddr = 10 (waddr1 on); 10 is binary for 2, so by the same 00→reg0 … 11→reg3 rule this aims at reg2 — which is dark right now. Set wdata = 1, then press ↻ clock write and follow the pulse: it runs along the shared clk line into all four AND gates at once, but only reg2's gate has its wen high, so only that one passes the edge through to reg2's clock. reg2 flips to 1 and starts glowing; the other three never saw an edge and hold. The decoder chose, the AND gate delivered the edge, the register remembered. Click any register to drill into its /dff.html — a one-bit register is exactly a D flip-flop.

why this matters

Every RISC-V instruction names registers. add x3, x1, x2 says "read x1, read x2, write x3" — that's exactly the two read ports and one write port you see here, each addressed by a register number. The real one just isn't shy about size: 32 entries of 32 bits instead of 4 entries of 1 bit.

Widen each register to 32 bits and grow the addresses to 5 bits (so the write side becomes a 5-to-32 decoder and each read port a 32-to-1 MUX) and you are looking at the literal heart of the CPU's datapath. The next page pairs this with the ALU — rdata and rdataB become the ALU's two operands — and the values stored here finally get something done to them.
