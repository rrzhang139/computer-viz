where we are: address-to-one-hot

A 2-bit binary number can point at one of 4 things. The decoder converts that 2-bit address into a one-hot signal: exactly one of sel0..sel3 goes high, the rest stay low. Toggle a1 and a0, watch the lit output move.

This is the smallest interesting routing primitive. Every register file, multiplexer, and instruction decoder uses one. It's how a CPU "picks which register to read."

the structure: 2 inverters + 4 ANDs

The body of the decoder is 2 inverters (left) that produce ~a1 and ~a0, and 4 three-input AND gates (right) — one per output. Each AND combines a unique polarity pattern of a1, a0, and EN.

Five vertical buses run between them: a1, ~a1, EN, a0, ~a0. Each AND taps three of the five — the right three for that AND's polarity pattern.

EN gates the whole thing

Click EN to toggle the enable. With EN=0, every AND's third input is 0 — so every output drops to 0, regardless of a1/a0. The decoder is silenced.

This is what makes the decoder useful in a register file: only the cycle where you want to write a register has EN=1, so only then does a sel line drive a write-enable wire. Other cycles, the register file is read-only.

walk the truth table

Cycle through all 4 addresses (00 → 01 → 10 → 11) and watch the lit sel move sel0 → sel1 → sel2 → sel3. Each combination is a unique polarity tap — the decoder is just 4 hardwired AND-of-polarities, one per address.

This scales: a 5-to-32 decoder (RISC-V's register-select) uses 5 inverters + 32 five-input ANDs. Same shape, more rows.

why this matters

The decoder is the bridge between a compact representation (a binary number) and a wire-level select (one wire per target). Computers store addresses compactly because they fit in registers; circuits route via per-target wires because that's what hardware can do.

Every place these two views meet — instruction decode, register-file read/write, memory address, branch target — there's a decoder shaped exactly like this one.
